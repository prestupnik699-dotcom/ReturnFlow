# PURPOSE

The source documentation (PROJECT_SPECIFICATION.md, ARCHITECTURE.md, DATABASE.md, API.md, ROADMAP.md, AI_RULES.md) was generated by an AI assistant rather than agreed upon with a live product owner, and contains a number of internal ambiguities and inconsistencies.

This document records every decision made to resolve such ambiguities during development, together with the reasoning behind each decision.

This document does not override PROJECT_SPECIFICATION.md, DATABASE.md, API.md, ROADMAP.md, or AI_RULES.md. It exists to fill gaps and resolve conflicts between them, following the source-of-truth priority order defined in AI_RULES.md Section 6.

Every new decision must be appended here before implementation begins. Never guess silently.

# DECISION LOG

### D-001 — Missing UI_UX.md and PROJECT_SPECIFICATION.md Part 2

Status: Accepted

These documents do not exist. UI details (colors, spacing, copywriting) will be proposed during Phase 4 (Application Foundation), following the design principles stated in PROJECT_SPECIFICATION.md Section 1.11-1.12.

---

### D-002 — Relationship between API.md and Supabase

Status: Accepted

API.md endpoints are treated as a logical service-layer contract, not literal HTTP routes.

- Simple CRUD operations -> direct Supabase JS SDK calls from `*.service.ts` files.
- PDF/Excel export, push notification triggers, complex statistics aggregation, and offline sync conflict resolution -> Supabase Edge Functions.

---

### D-003 — Role Permission Matrix

Status: Deferred

Roles (`Owner`, `Administrator`, `StoreManager`, `Receiver`, `Employee`, `Viewer`) are named in DATABASE.md but their permissions are not defined. A permission matrix will be drafted and approved before Phase 3 (Authentication / RLS policies) is implemented.

---

### D-004 — Organization Membership Model

Status: Accepted

The database supports multi-organization membership per user (as modeled by the `memberships` table). The UX presents a single active organization/store at a time, with a switcher, rather than exposing multiple organizations simultaneously.

---

### D-005 — Offline Storage Technology

Status: Accepted

`expo-sqlite` is used for local persistence and the offline sync queue. Rationale: the application must support offline search and filtering across 10,000+ `return_items` records, which a simple key-value store (e.g. MMKV) cannot support efficiently. `expo-sqlite` works in the Expo managed workflow without a custom dev client.

---

### D-006 — Conflict Resolution Policy

Status: Accepted (subject to refinement in Phase 14)

Default policy: last-write-wins based on `updated_at`.

Exception: once `return_items.status` reaches a terminal state (`returned` or `archived`), the server value takes precedence over a conflicting client update, to prevent an offline device from silently reverting a completed workflow.

---

### D-007 — `return_items.status` vs `priority` vs `deleted_at`

Status: Accepted

- `deleted_at IS NOT NULL` is the single source of truth for soft-delete. The `deleted` value is removed from the `status` enum to avoid two representations of the same fact.
- `status` enum becomes: `pending`, `urgent`, `returned`, `archived`.
- `urgent` is a workflow state (set manually by a user, or automatically by a background job based on `app_settings.urgent_days`), independent from `priority`, which reflects business importance (`low`, `normal`, `high`, `critical`).
- Archive/Restore operations are implemented via `PATCH /returns/:id` (status change), not as separate endpoints, consistent with API.md not listing dedicated archive/restore routes.

---

### D-008 — Duplicated Settings Fields (`organizations` vs `app_settings`)

Status: Accepted

`organizations` is the single source of truth for organization-level branding: `default_language`, `primary_color`.

The duplicate fields `default_language` and `theme_color` are removed from `app_settings`. `app_settings` retains only operational/behavioral configuration: `urgent_days`, `allow_exports`, `allow_chat`, `allow_comments`.

---

### D-009 — `suppliers.favorite` Scope

Status: Accepted

Implemented literally as specified in DATABASE.md: a single boolean column on the `suppliers` table, meaning "favorite" is organization-wide, not per-user.

---

### D-010 — Chat "Announcements" Room Type vs `announcements` Table

Status: Accepted

The `announcements` table and its dedicated feature/endpoints are the working implementation of announcements in Version 1.0.

The `Announcements` value in the `chat_rooms.type` enum is retained in the schema for forward compatibility but is not used to build a separate feature in Version 1.0.

---

### D-011 — Missing `updated_at` / `deleted_at` Columns

Status: Accepted

DATABASE.md's general principle ("Every table: Created At, Updated At, Soft Delete...") takes precedence over the specific per-table column lists, which appear to have been omitted inconsistently by the documentation's original author.

`updated_at` and `deleted_at` are added to: `profiles`, `memberships`, `announcements`, `notifications`.

---

### D-012 — Target Expo SDK Version

Status: Accepted

Project targets **Expo SDK 57**. Chosen based on the Expo Go version actually installed on the developer's physical test device at project start (July 2026), since Expo Go on app stores updates independently of this project.

If Expo Go's required SDK changes again after real application code exists, the project should be upgraded to match using Expo's official SDK upgrade guide rather than recreated from scratch.

---

### D-013 — Email Confirmation Disabled in Development

Status: Accepted (must be reverted before production)

`Confirm email` is disabled in Supabase Auth provider settings for the development project (`returnflow-dev`), to allow faster iteration without needing to process confirmation links/deep links this early in development.

Must be re-enabled before any production release, per DEPLOYMENT.md pre-release checklist.

---

### D-014 — Production Supabase Environment Deferred

Status: Accepted

Only one Supabase project (`returnflow-dev`) exists at this stage. Separate STAGING and PRODUCTION Supabase projects (per DEPLOYMENT.md environments section) will be created closer to release, once the schema and core features stabilize.

### D-015 — profiles.status vs deleted_at

Status: Accepted

Same reasoning as D-007. `deleted_at` is the single source of truth for deletion. `profile_status` enum contains only `active`, `vacation`, `blocked` — `deleted` is intentionally omitted.

---

### D-016 — memberships.store_id Nullability

Status: Accepted

`store_id` on `memberships` is nullable. `NULL` means organization-wide access (typical for `Owner` / `Administrator`). A non-null value scopes the membership to a single store (typical for `StoreManager`, `Employee`, `Receiver`, `Viewer`).

### D-017 — notifications.type Enum Values and chat_rooms.store_id Scoping

Status: Accepted

`notifications.type` was left undefined in DATABASE.md. Values chosen to match the feature set already defined elsewhere in the documentation: `return_created`, `return_urgent`, `return_returned`, `comment_added`, `chat_message`, `announcement`, `invitation`.

`chat_rooms.store_id` is nullable, following the same pattern as `memberships.store_id` (D-016): `NULL` means an organization-wide room (e.g. General), a non-null value scopes the room to one store (e.g. a store's Returns chat).

### D-018 — Role Permission Matrix (resolves D-003)

Status: Accepted

### D-019 — Missing `invitations` Table

Status: Accepted

DATABASE.md's 18-table list does not include a table to store invitation codes, even though API.md requires an invitation code for registration and ROADMAP.md Phase 7 lists "Invite User" / "Accept Invitation" as features.

A new `invitations` table is added: `organization_id`, `store_id` (nullable, same semantics as `memberships.store_id`), `role`, optional `email`, a short human-typeable `code`, `status` (`pending`/`accepted`/`revoked`/`expired`), `invited_by`, `expires_at`, `accepted_at`.

Accepting an invitation is handled by a dedicated `accept_invitation(code)` SQL function (security definer), rather than opening up the `memberships` table's INSERT policy to any newly-signed-up user. This keeps the existing membership-management policies (D-018) unchanged and narrow.
